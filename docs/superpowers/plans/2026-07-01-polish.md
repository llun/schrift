# Polish Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement design spec Phase 10 (the final phase of the original 10-phase build sequence) — empty states, loading states, error states, and pull-to-refresh, closing concrete gaps found by reading the current Home/Editor/Connect code rather than guessing what "polish" should mean.

**Architecture:** Every change here was validated end-to-end against this machine's Xcode 26.6/iOS 26.5 toolchain in a scratch project before being written into this plan — including Simulator screenshots of all three new UI states — and each item traces to a concrete, verified gap in the current code, not a speculative addition:

- **`HomeViewModel.load()` and `EditorViewModel.load()` both already say "Pull to refresh to try again." in their error messages, but neither `DocumentListView` nor `EditorView` had a `.refreshable` modifier — the error text was promising a gesture that didn't exist.** Verified directly by grepping both files for `refreshable` before writing this plan (zero matches). Fixed by adding `.refreshable { await viewModel.load() }` to `DocumentListView`'s `ScrollView` and to `EditorView`'s read-only-mode `ScrollView` (not the `TextEditor` in editing mode, where pull-to-refresh doesn't apply).
- **Zero-result states were rendering as blank space, not a message** — `DocumentListView`'s `documentSection(title:documents:)` already guards on `if !documents.isEmpty`, so a genuinely empty pinned/recent list (or a search with no matches) silently showed nothing below the search bar/segmented control. `EditorView`'s block list had the same gap for a document with no content. Fixed with native `ContentUnavailableView` — `ContentUnavailableView.search(text:)` for empty search results (matching the same component `HomeSplitView`'s "Select a Document" state already uses, from the iPad Adaptive Layout plan), and a custom `ContentUnavailableView("No Documents", ...)` / `ContentUnavailableView("Empty Document", ...)` for the other two cases.
- **A real bug caught during scratch validation, not by a failing test: the new empty-state views must not render simultaneously with an existing error banner.** `HomeViewModel.load()` on failure leaves `pinnedDocuments`/`recentDocuments` empty (they're never populated in the `catch` branch) and `EditorViewModel.load()` on failure leaves `blocks` empty the same way — so a naive `if documents.isEmpty { ContentUnavailableView(...) }` would show the new empty-state message stacked directly underneath the existing red error text on every failed load, which is confusing double-messaging. Verified this by building the naive version first and reading the code path, before ever taking a screenshot. Fixed by gating every new empty-state branch on `viewModel.errorMessage == nil`, so a failed load falls through to the pre-existing (and already-correct) error-banner-only behavor, and the new empty-state UI only appears for a genuinely successful, genuinely empty load.
- **`ConnectViewModel.handleLoginComplete()` runs a real network call (`GET users/me/`) with no loading indicator** — after the WebView login sheet dismisses, the user sees nothing happen until the call resolves (success navigates away, failure shows an error). Fixed with a new `isSigningIn: Bool` on `ConnectViewModel`, set for the duration of `handleLoginComplete()` via `defer` (the same guaranteed-reset idiom already used by `OptionsViewModel.duplicate()` from the Options Sheet plan), and `ConnectView` disables the Sign In button (`DocsButton` already has a built-in `isDisabled` parameter that both restyles and calls `.disabled()`) and the recent-servers list, showing a `ProgressView` beneath the button while signing in.
- **Empty search results guard `.errorMessage == nil` too** — `HomeViewModel.search()` on failure leaves `searchResults` at its prior value (often already empty) without clearing it, so the same double-messaging risk applies to the search branch; handled with the identical guard.

**Tech Stack:** Swift 6.0, SwiftUI, XCTest, XcodeGen 2.45 (Homebrew), Xcode 26.6 / iOS 26.5 SDK, deployment target iOS 18.0.

## Global Constraints

- Deployment target: iOS 18.0, universal app.
- Zero third-party Swift package dependencies.
- `project.yml` is the single source of truth; regenerate via `xcodegen generate` after adding any new file, **before** building/testing.
- Verified local build/test destination: `-destination 'platform=iOS Simulator,name=iPhone 17'`.
- Each task ends in its own commit.
- A benign toolchain warning — `warning: Metadata extraction skipped. No AppIntents.framework dependency found.` — appears in every build regardless of code changes. Ignore it.
- Every new `ContentUnavailableView` branch (Home's "No Documents", Home's search-empty, Editor's "Empty Document") **must** be gated on `viewModel.errorMessage == nil` — do not show the new empty-state UI at the same time as the existing error banner text.
- `ConnectViewModel.isSigningIn` must be reset via `defer`, not by setting it `false` at the end of the function body — matching the plan's own verified-safe idiom and guaranteeing the flag clears even if `sessionStore.signIn(serverURL:)` throws.
- Do not add a `.refreshable` modifier to `EditorView`'s `TextEditor` (editing mode) — pull-to-refresh only applies to the read-only rendering `ScrollView`.
- This plan has no new XCTest files by design — every change is UI-state glue (conditional view branches, a boolean loading flag toggled around one already-tested network call) verified by build-check and Simulator screenshots, matching the established pattern for UI-wiring tasks in every prior plan (Home Screen, Editor Screen, Share Sheet, Options Sheet, iPad Adaptive Layout). If `RootView.swift` is temporarily swapped for screenshots, it MUST be reverted to the real auth-gated version before committing — never commit a temporary version.

        isPresentingWebLogin = false
        guard let serverURL = pendingServerURL else { return }

        struct Me: Decodable {}
        let client = apiClientFactory(serverURL)
        do {
            let _: Me = try await client.get("users/me/")

```swift
    func handleLoginComplete() async {
        isPresentingWebLogin = false
        guard let serverURL = pendingServerURL else { return }

        struct Me: Decodable {}
        let client = apiClientFactory(serverURL)
        do {
            let _: Me = try await client.get("users/me/")
            try sessionStore.signIn(serverURL: serverURL)
            recentServers.addServer(serverURL)
        } catch {
            errorMessage = "Sign-in could not be confirmed. Please try again."
        }
    }
```
